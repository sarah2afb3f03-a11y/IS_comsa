## 1. Транзакции

**Транзакция** - группа операций, которые должны выполниться как единое целое. Либо все операции проходят успешно (`COMMIT`), либо система полность откатывает все изменения назад (`ROLLBACK`).
Промежуточные состояния внутри последовательности операций **не видны** другим транзакциям. 

**ACID** - это набор принципов, которые обеспечивают надежность и целостность данных в базе данных. 
* A (Атомарность) - гарантирует, что результаты работы транзакции не будет зафиксированы в системе частично. Будут выполнены все операции транзакции, либо не выполнено ни одной.
* C (Согласованность) - после выполнения транзакции база данных должна быть в согласованном состоянии. Во время выполнения транзакции согласованность не требуется. Например, уникальность значений, проверка на положительные числа и т.д.).
* I (Изоляция) - во время выполнения транзакции другие транзакции (выполняющие параллельно) не должны оказывать влияние на результат транзакции.
  - Виды конфликтов при параллельном доступе к данным
    
    При организации параллельного доступа к данным могут возникнуть различные проблемы:
      * Грязное чтение («dirty read»)
        - Проблема возникает, если одна транзакция видит измененные данные другой (незавершенной транзакции).
      * Неповторяющееся чтение («nonrepeatable read»)
        -  Проблема возникает, если один и тот же запрос в рамках транзакции возвращает разные результаты
      * Фантомное чтение («phantom read»)
        - Проблема возникает, если один и тот же запрос в рамках транзакции возвращает разное число записей
      * Аномалии сериализации
        - Проблема возникает, когда результат параллельного выполнения транзакций отличается от любого возможного последовательного (serial) выполнения этих же транзакций.
        - Проще говоря: Если бы транзакции выполнялись по одной, результат был бы один. Но из-за одновременной работы получается другой результат.
    
    Read Uncommitted — не поддерживается PostgreSQL.
  - Уровни изоляции:
    * **Read Uncommitted** - позволяет читать незафиксированные данные из других транзакций, что может привести к "грязному чтению".
    * **Read Committed** - позволяет читать только зафиксированные данные, что предотвращает "грязное чтение", но может привести к "фантомному чтению".
    * **Repeatable Read** - гарантирует, что в рамках одной транзакции данные не изменятся, даже если другие транзакции их изменят.
    * **Serializable** - самый строгий уровень изоляции, который полностью изолирует транзакции, предотвращая любые аномалии.
* D (Устойчивость) - при успешном завершении транзакции результаты ее работы должны остаться в системе независимо от возможных сбоев оборудования, системы и тд.

**Структура транзакции**
* `BEGIN`- начало транзакции
* `COMMIT` - сохранение в БД
* `ROLLBACK` - полностью отменяет все изменения с момента `BEGIN`
* `SAVEPOINT name` - сохраняет промежуточную транзакцию. 
* `ROLLBACK TO name` - откатил до промежуточной транзакции.

**Какими бывают транзакции?**
* Неявные (implicit) транзакции - это транзакции, которые СУБД создаёт автоматически, без команды `BEGIN`. Просто пишем SQL-запрос и самостоятельно оборачивается в `BEGIN`. Один SQL-запрос = 1 транзакция. После выполнения происходит автокоммит. 
* Явные (explicit) транзакции - транзакции, которые задает пользователь самостоятельно (`BEGIN`, `COMMIT`).

## 2. Время и транзакции

Есть несколько функции определяющие время:
* `CURRENT_TIME` - выводит время. В транзакции будет показывать всегда время начало транзакции. 
* `CURRENT_TIMESTAMP` - выводит дата + время. В транзакции будет показывать всегда время начало транзакции.
* `now()` - выводит дата + время. В транзакции будет показывать всегда время начало транзакции.
* `transaction_timestamp()` - выводит дата + время. В транзакции будет показывать всегда время начало транзакции.
* `statement_timestamp()` - Время начала текущего SQL-запроса. В транзакции он будет меняться. 
* `clock_timestamp()` - Реальное текущее время. Неважно в явной или неявной транзакции.

В чем разница между `CURRENT_TIMESTAMP`, `now()`, `transaction_timestamp()` ? 

В них нет никакого отличия по факту. `now() = CURRENT_TIMESTAMP = transaction_timestamp()`. Просто разные написания для понимания к чему они идут.  

## 3. Идентификация транзакций

**Идентификатор транзакции (xid)** — назначается СУБД для каждой новой транзакции;
`xid` — уникален. 
* уникален в пределах жизненного цикла базы
* выдаётся по порядку (почти всегда увеличивается)
* используется для определения "кто старше, кто младше". Чем больше `xid`, тем позже оно создано. **НО ЕСТЬ НЮАНС.**

При создании/модификации записи хранится `xid` транзакции.

**Важно:**
- `xid` — это 32-битное число (беззнаковое), и оно может переполняться (`wraparound`).
Поэтому существует механизм `autovacuum`.
- `xid` — начинается с 3 (значения 0, 1, 2 — зарезервированы).

**xid wraparound problem**

После достижения максимума счётчик переполняется и снова становится маленьким.

Проблема:
-  PostgreSQL определяет "кто старше, кто младше" по xid.
-  Если старые строки не обработать, система может перепутать старое и новое.

Это и есть `xid wraparound problem`

1. **FROZEN** - Установка статуса неактуальных записей. 
  
    Статус FROZEN — для очень старых транзакций.
  
    Когда строка очень старая и точно закоммичена:
  
    `xmin → 2 (FrozenTransactionId)`
  
    2 — это специальный xid, означающий:
  
    эта строка настолько старая, что её можно считать вечной
  
    После этого:
  
    сравнивать её xid больше не нужно
  
    переполнение ей не страшно
  
    Это делает `VACUUM / autovacuum`.

2. **Экономия xid**

    Если транзакция ничего не меняет — ей выдаётся виртуальный xid.
    
    Если бы каждая транзакция получала реальный xid:
    
      *  счётчик рос бы очень быстро
      *  быстрее наступал бы wraparound
    
    Поэтому:
      *  BEGIN → только virtual xid
      *  SELECT → всё ещё virtual
      *  UPDATE/INSERT → выдаётся реальный xid
   
    Это замедляет расход xid.

3. **WARNING: database must be vacuumed within N transactions**

    Когда база приближается к опасной зоне, появляется сообщение:

    PostgreSQL считает: `age(datfrozenxid)`
   
    Если возраст близок к лимиту: `autovacuum_freeze_max_age` (по умолчанию ~200 млн)

    он предупреждает: скоро будет опасно

4. **ПАНИКА (самое важное)**

    Если проигнорировать предупреждение:
    
   PostgreSQL блокирует обычные операции: `ERROR: database is not accepting commands to avoid wraparound data loss`
    
    Разрешается только: `VACUUM`
    
    Это аварийная защита от потери данных.

- `xid = 0` - InvalidTransactionId
  
    Это специальное значение:
    -  «Нет транзакции» / «не назначено». Используется внутри системы.
    
    Например:
    -  если строка ещё не имеет xmax
    -  если xid не был присвоен
  
- `xid = 1` - BootstrapTransactionId
  
    Используется при инициализации кластера:
      `initdb`
  
    Когда создаются системные каталоги (pg_class, pg_attribute и т.д.), они создаются под транзакцией с `xid = 1`. Это "загрузка системы".

- `xid = 2` - FrozenTransactionId

    Очень важное значение.
  
    Означает:
    - Эта строка настолько старая, что считается видимой для всех транзакций.
    -  Используется VACUUM.
    
    Когда строка "замораживается": `xmin → 2`
    
    Это значит:
    - её создала древняя транзакция
    - её уже не нужно сравнивать
    - она безопасна при `wraparound`

**Скрытые (системные) поля**
* `xmin` — хранит `xid` транзакции, в рамках которой запись (копия записи) была создана
* `xmax` — хранит `xid` транзакции, в рамках которой запись была удалена
* `cmin` — идентификатор команды, создавшей запись
* `cmax` — идентификатор команды, удалившей запись

## 4. Реализация транзакций в PostgreSQL

Реализация транзакций (DML) в PostgreSQL — MVCC

MVCC - многоверсионное управление конкурентным доступом

**Главная идея:** каждое изменение строки создаёт новую версию строки, а старая версия не удаляется сразу.

**Это позволяет:**
- читать данные без блокировок
- выполнять много транзакций параллельно

**Что значит Multi-Version?**

В PostgreSQL одна строка может иметь несколько версий.

| xmin | xmax | client_id | accvalue |
| ---- | ---- | --------- | -------- |
| 100  | 105  | 1         | 5000     |
| 105  | 0    | 1         | 7000     |

в PostgreSQL: Serializable Snapshot Isolation (SSI).

**Особенности MVCC (SSI):**
* при изменении данных в транзакции — создается новая копия данных;
* существующие копии данных не изменяются;
* при выборке данных берется подходящая для данной транзакции версия данных;

## 5. Изоляция транзакций

- Во время выполнения транзакции параллельные транзакции не должны оказывать влияния на её результат.
- Полноценная реализация требует очень много ресурсов!
- В реальных системах используется концепция уровней изолированности транзакций.

**Выбор режима изоляции транзакции**
- `BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;`
```
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

**Snapshot Isolation (SI)**

**Snapshot** - это моментальный снимок состояния базы данных.

Когда начинается транзакция, PostgreSQL говорит:
- Вот список транзакций, которые уже завершились
- Вот список транзакций, которые ещё выполняются

На основе этого строится правило видимости строк.
- Каждая транзакция видит свою версию базы
- И это не копия таблицы, а набор правил видимости.

Предоставляется мэнеджером транзакций:
- READ COMMITTED: свой снимок для каждого выполняемого SQL-предложения.
    - Каждый SQL-запрос получает новый snapshot.
    - Поэтому можно увидеть изменения других транзакций.
- REPEATABLE READ, SERIALIZABLE: снимок — при выполнении только первого SQL-предложения.
    - Snapshot создаётся один раз — при первом SQL.
    - Поэтому изменения других транзакций не видны.

Выбор подходящей версии записи определяется по правилам проверки видимости, которые используют:
- снимок;
- значения `xmin`, `xmax`, `cmin`, `cmax`, ...

**Что именно хранит snapshot?**

Snapshot содержит:
- xmin
- xmax
- список активных xid

**Как посмотреть snapshot?**
- `SELECT pg_current_snapshot();`
- `SELECT txid_current_snapshot();`
